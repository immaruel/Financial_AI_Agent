# 06. Risk Controller and Answer Composer Agent

> **Implementation alignment note**: In the current implementation, `HypothesisCheckerAgent` and `RiskControllerAgent` are separate classes. `HypothesisCheckerAgent` validates whether claims are grounded in graph evidence, while `RiskControllerAgent` post-processes the final answer for investment phrasing, overconfidence, and warning messages. This document keeps those roles separate, while describing them together from the perspective of end-user safety.

---

## Part 1: Risk Controller Agent

### 1. Purpose

Validate the answer draft generated by Answer Composer against two categories of risk, then populate `risk_warnings`.

- **Hypothesis-verification linkage**: interpret the answer draft together with the `HypothesisCheckerAgent` verdict to determine whether the output is truly grounded in graph evidence. This includes temporal order, causal overreach, conflicting evidence, and confusion between rumor and filing.
- **Risk filter, the Risk Controller role**: act as the last safety layer that prevents investment recommendations or guaranteed-profit language from remaining in the final answer.

---

### 2. Inputs / Outputs

**Input:**

| Item | Description |
|------|-------------|
| draft `StructuredAnswer` | Draft generated by Answer Composer |
| `HypothesisChecker` verdict | Validation result for unsupported claims and contradictions |
| `AgentConfig.prohibited_phrases` | Forbidden-phrase dictionary |

**Output:**

Revised `StructuredAnswer` with `risk_warnings` filled and risky wording cleaned for final delivery

---

### 3. Processing Logic

Validate the following five items in sequence.

#### 3.1 Detect Investment Recommendation Phrases

```python
prohibited_phrases = [
    "will definitely rise", "guaranteed profit", "invest now", "buy recommendation",
    "sell recommendation", "profit guaranteed", "loss-free",
]
```

If the generated text contains any of these patterns, remove the sentence and add a warning.

#### 3.2 Detect Confusion Between Confirmed and Speculative Language

Inspect each event's `certainty`.

```text
certainty == "speculated" and the answer uses definitive language → add warning
certainty == "disclosed"  and the answer uses tentative language  → add warning
```

#### 3.3 Add Staleness Warning

If event times fall outside the requested time window defined by `time_constraints["window_days"]`.

```python
if chain.events:
    latest_time = max(e.get("event_time") for e in chain.events if e.get("event_time"))
    if (now - datetime.fromisoformat(latest_time)).days > window_days * 1.5:
        warnings.append("The collected events are older than the requested time window. Please verify the latest information.")
```

#### 3.4 Force Presentation of Both Sides When Conflicting Evidence Exists

If Evidence Retriever detected `conflicts`.

```python
for ev_result in evidence_results:
    if ev_result.conflicts:
        warnings.append(
            f"Conflicting evidence exists for event {ev_result.event_id}: "
            + "; ".join(ev_result.conflicts)
        )
```

#### 3.5 Detect Unsupported Causal Leap

If `HypothesisCheckerAgent` reports unsupported inference or contradiction, and the answer still preserves definitive causal language, add a warning.

---

### 4. Design Rationale

**Why is a post-processing filter necessary?**  
If every safety rule lives inside Answer Composer, generation logic and validation logic become entangled, which makes rule changes harder. By keeping the filter separate, forbidden phrases, low-confidence policy, and risk warnings can be managed independently.

---

## Part 2: Answer Composer Agent

### 1. Purpose

Integrate `QueryPlan`, `SubGraphResult`, `EvidenceResult`, `CausalChain`, and `risk_warnings` into the final structured answer, `StructuredAnswer`.

This is the **only point in the agent pipeline where the LLM, Qwen, is called**.

---

### 2. Inputs / Outputs

**Input:**

| Item | Description |
|------|-------------|
| `QueryPlan` | Original query, intent, and time constraints |
| `SubGraphResult` | Retrieved nodes and edges |
| `List[EvidenceResult]` | Source evidence |
| `CausalChain` | Time-ordered event chain |
| `List[str]` | `risk_warnings` |
| `QwenLLMClient` | LLM used for answer generation |

**Output:**

`StructuredAnswer` (`utils/schemas.py`)

| Field | Type | Description |
|------|------|-------------|
| `summary` | str | Core summary, one to three sentences |
| `timeline` | List[Dict] | Timeline of key events |
| `related_companies` | List[str] | List of related companies |
| `impact_analysis` | str | Description of beneficiary or loss logic |
| `counter_evidence` | str | Contradictory or opposing evidence |
| `confidence` | float | Overall answer confidence |
| `sources` | List[Dict] | Structured list of evidence sources |
| `risk_warnings` | List[str] | Warning list from Risk Controller |

---

### 3. Processing Logic

#### 3.1 Build the LLM Prompt

```text
[System Prompt]
You are an expert analyst of financial news and filings.
Generate a fact-based answer using the given graph data and supporting documents.
Never use investment recommendation or guaranteed-return language.

[User Prompt]
Query: {original_query}

[Related Events]
{format causal_chain.events in chronological order}

[Supporting Documents]
{include top N passages from evidence_results}

[Related Companies]
{company list derived from seed_nodes}

Based on the information above, answer in the following format:
- Core summary
- Event timeline
- Impact analysis
- Explicit note on areas with insufficient evidence
```

#### 3.2 Build the Timeline

Constructed by rules, without LLM generation.

```python
timeline = [
    {
        "date": event.get("event_time", ""),
        "event_type": event.get("event_type", ""),
        "description": event.get("trigger_text", ""),
        "company": event.get("subject_entity_id", ""),
        "certainty": event.get("certainty", ""),
        "sources": [ev for ev in sources if ev["event_id"] == event["id"]],
    }
    for event in causal_chain.events
]
```

#### 3.3 Extract Related Companies

Pull from Company nodes inside `SubGraphResult`.

```python
related_companies = [
    node["name"]
    for node in subgraph.nodes
    if node.get("label") == "Company"
]
```

#### 3.4 Calculate Confidence

```python
confidence = (
    causal_chain_avg_confidence * 0.6 +
    evidence_coverage_ratio * 0.4  # ratio of events that have evidence
)
```

---

### 4. Design Rationale

**Why build the timeline with rules instead of an LLM?**  
The graph already contains a time-sorted event list. If the LLM constructs the timeline freely, it may reorder events or invent events that do not exist. Rule-based construction guarantees timeline accuracy while keeping the LLM focused on narrative text such as `summary` and `impact_analysis`.

**Why keep `sources` as a separate field?**  
In finance, every claim should expose its source. Keeping `sources` structured makes it easy for the user interface to display links such as "This information was verified in filing X."
